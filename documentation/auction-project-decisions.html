Fullstack TypeScript Fish Auction House

This document captures the requirements from .github\copilot-instructions.md and the decisions made during the grill-me design session.

  • TanStack Start
  • TanStack Router
  • TanStack Query
  • PostgreSQL
  • Drizzle ORM
  • Server-Sent Events
  • Fish auctions
  • pnpm
  • oxlint
  • oxfmt

Original product requirements

  • Build an auction house for selling and buying items.
  • Manage an inventory of items for sale.
  • Let users add new items for sale, place bids on items, and complete sales.
  • Provide a dashboard for current auctions, bids, and sales.
  • Provide auction statistics including total sales, average bid price, and most popular items.
  • Use modern TypeScript ecosystem best practices and showcase TypeScript as a fullstack language.
  • Include metrics and monitoring for performance and errors.
  • Include a way to generate new items, bids, and sales.
  • Ensure that when a buyer places a bid, the visible bid chain updates in real time for all users.
  • Items are fish, and fish amounts are measured in kilograms.
  • Use TanStack Start as the base, with TanStack Router, TanStack CLI, and TanStack Query.
  • Use pnpm for package management.
  • Use oxlint and oxfmt for linting and formatting, configured according to TanStack recommendations.

Implementation strategy

The first milestone is a full vertical auction slice rather than a tooling-only scaffold or analytics-first build.

The initial deliverable should be a runnable TanStack Start app with tooling, PostgreSQL and Drizzle schema, seeded demo users/fish/auctions, and the live auction detail flow. Dashboards, analytics, metrics, and the simulator are layered on top once the auction flow works end to end.

Resolved decisions

Project baseline Use the TanStack CLI scaffold first, then add domain modules around TanStack Start conventions.
Package manager Use pnpm.
Linting and formatting Configure oxlint and oxfmt with required scripts for lint, format, format check, and tests.
Persistence Use PostgreSQL with Drizzle ORM.
Local infrastructure Use Docker Compose for PostgreSQL and run the app/scripts with pnpm.
Migrations Use Drizzle Kit migrations committed to the repository, with pnpm scripts for db:generate, db:migrate, db:seed, and db:studio.
Domain model Separate fish_items from auctions. Fish inventory is distinct from auction lifecycle.
Fish weight Input and display kilograms in the UI, but store integer grams in the database.
Money Store monetary values as integer cents with one configured currency.
Auction type Use ascending English auctions with a configurable minimum bid increment.
Bid consistency Enforce bids inside a database transaction, reject stale bids, and broadcast only after commit.
Real time Use Server-Sent Events per auction for bid and sale updates.
Meaning of chain Treat "chain" as the real-time visible bid history, not blockchain.
Users Use seeded demo users with a header demo user switcher instead of full auth in the MVP.
Permissions Sellers manage their own fish and auctions, buyers bid on other sellers' active auctions, sellers cannot bid on their own auctions, and admins can close, withdraw, and view all records.
Sales completion Auctions can be closed manually or by time. The highest valid bid wins; auctions with no bids close as unsold.
Inventory states Use draft, listed, in_auction, sold, and withdrawn.
Auction timing Use startsAt and endsAt. Only active auctions accept bids. Finalization can be lazy or admin-triggered.
Data access Use TanStack Start server functions for core queries and mutations, TanStack Query for client cache, and a small HTTP/SSE surface for streaming and simulator access.
Validation Use shared Zod schemas across server functions, API handlers, and UI forms.
Statistics Compute live relational aggregates from PostgreSQL with appropriate indexes, not a separate analytics store.
Monitoring Use OpenTelemetry, a Prometheus-style metrics endpoint, and structured logs.
Testing Use Vitest unit tests and integration tests. Do not add Playwright for now.
Realtime testing Integration-test the SSE/event broadcaster at the server level with multiple listeners receiving the same ordered bid event.

Fish inventory model

Required fish fields for the MVP:

  • Species
  • Display name
  • Weight in kilograms in the UI, stored as grams
  • Catch region
  • Freshness or grade
  • Starting price
  • Seller
  • Optional description
  • Optional external image URL

Species should be selected from a curated MVP list so popularity statistics group cleanly: salmon, cod, tuna, halibut, mackerel, trout, and herring.

Dashboards and routes

Routes

  • / - live operational dashboard
  • /auctions/$auctionId - auction detail page with live bid history
  • /inventory/new - add a fish item
  • /admin - history, admin, and statistics dashboard
  • /metrics - scrapeable metrics endpoint

Live operational dashboard

Show active auctions, latest bids, recently completed sales, and inventory requiring action.

History/admin dashboard

Focus on audit and management: completed and unsold auctions, sale records, bid history, inventory status changes, and admin actions such as closing auctions or withdrawing fish items.

Statistics should live in this dashboard as summary cards and charts with filters by date range and fish species.

Statistics definitions

  • Total sales: aggregate completed sale value from accepted winning bids.
  • Average bid price: average accepted bid amount, stored and computed in cents.
  • Most popular fish items: rank fish species by bid count, while also showing total kilograms sold and total sales value.

Realtime behavior

SSE events should be explicit and typed. Supported event shapes should include bid.accepted, bid.rejected for the actor only, auction.closed, and sale.completed.

Event payloads should include the auction id, bid id, amount in cents, bidder display name, timestamp, and a current highest-bid snapshot. The auction detail page should patch the TanStack Query cache immediately from accepted bid events, while aggregate dashboard and statistics queries should be invalidated and refetched.

Bid validation and errors

The server should return explicit typed bid errors for:

  • Stale bid
  • Auction not active
  • Seller bidding on their own auction
  • Insufficient increment
  • Invalid amount

Invalid bids must not be silently accepted or hidden behind generic refresh behavior.

Generated demo activity

Provide deterministic seed scripts and an optional local simulator. The simulator should create auctions and bids through the same server APIs used by the application so validation, persistence, metrics, and realtime updates are exercised.

Expose commands such as pnpm db:seed and pnpm simulate. The simulator should accept options for auction count, bid interval, and duration, with deterministic fake fish data as the default.

Monitoring and metrics

The first monitoring pass should include OpenTelemetry instrumentation, structured server logs, and a Prometheus-style metrics endpoint.

Mandatory domain metrics:

  • Accepted bids counter
  • Rejected bids counter
  • Auctions created counter
  • Auctions closed counter
  • Sales completed counter
  • Total sale value in cents
  • Request latency histogram
  • Bid mutation duration histogram

Testing scope

The test plan is Vitest-focused. Use unit tests for domain rules and integration tests for server functions, database behavior, and realtime broadcaster behavior.

Playwright/browser automation is intentionally out of scope for now.

Deletion and audit policy

Do not hard-delete auctions, bids, or sales in the MVP. Use statuses such as withdrawn, closed, and unsold so the admin dashboard and statistics remain auditable.
