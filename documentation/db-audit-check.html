Security audit follow-up

Database audit check

This page documents the database hardening steps identified during the audit: adding PostgreSQL constraints for core auction invariants and splitting migration privileges from application runtime privileges.

1. Add database constraints for core invariants

Application validation already protects normal UI and server-function flows, but database constraints make the invariants durable. They prevent bad data from direct SQL scripts, future imports, skipped validation paths, or application bugs.

Table Constraint Reason
fish_items weight_grams > 0 Prevents impossible inventory weights.
fish_items starting_price_cents > 0 Prevents free or negative starting prices.
fish_items, sales currency = 'NOK' Enforces the current single-currency decision.
auctions starts_at < ends_at Prevents invalid auction windows.
auctions minimum_increment_cents > 0 Prevents zero or negative bid increments.
bids, rejected_bids, sales amount_cents > 0 Keeps bid and sale records financially valid.

Example migration SQL

ALTER TABLE fish_items
  ADD CONSTRAINT fish_items_weight_positive CHECK (weight_grams > 0),
  ADD CONSTRAINT fish_items_starting_price_positive CHECK (starting_price_cents > 0),
  ADD CONSTRAINT fish_items_currency_nok CHECK (currency = 'NOK');

ALTER TABLE auctions
  ADD CONSTRAINT auctions_window_valid CHECK (starts_at < ends_at),
  ADD CONSTRAINT auctions_minimum_increment_positive CHECK (minimum_increment_cents > 0);

ALTER TABLE bids
  ADD CONSTRAINT bids_amount_positive CHECK (amount_cents > 0);

ALTER TABLE rejected_bids
  ADD CONSTRAINT rejected_bids_amount_positive CHECK (amount_cents > 0);

ALTER TABLE sales
  ADD CONSTRAINT sales_amount_positive CHECK (amount_cents > 0),
  ADD CONSTRAINT sales_currency_nok CHECK (currency = 'NOK');
Add constraints through Drizzle migrations and check existing seed/demo data before applying them. If old rows violate a constraint, clean the data first rather than adding a looser rule.

2. Split migration and runtime database roles

The application should not run with the same PostgreSQL role that owns the schema and applies migrations. A compromised app process should not be able to drop tables, alter constraints, create privileged functions, or grant itself access.

Role Permissions Used by
auction_migrator Owns schema objects and can create, alter, drop, and migrate database objects. Drizzle Kit migration commands and CI/deploy migration step.
auction_app Can connect, use the public schema, and perform only required SELECT, INSERT, and UPDATE operations. The TanStack Start application at runtime.

Example role setup SQL

CREATE ROLE auction_migrator LOGIN PASSWORD 'replace-with-secret';
CREATE ROLE auction_app LOGIN PASSWORD 'replace-with-secret';

GRANT CONNECT ON DATABASE auction TO auction_app;
GRANT USAGE ON SCHEMA public TO auction_app;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO auction_app;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO auction_app;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE ON TABLES TO auction_app;

Use the app role in the runtime DATABASE_URL. Use a separate migration-only URL for Drizzle Kit, local migration commands, and deployment automation.

Implementation checklist

  1. Generate a Drizzle migration for the new constraints.
  2. Run the migration against local seed data and fix any invalid rows before merging.
  3. Create separate PostgreSQL roles for migration and runtime access.
  4. Update local and deployment environment variables to use the correct role per task.
  5. Confirm the app role cannot run DDL such as ALTER TABLE or DROP TABLE.
  6. Keep future migrations owned by the migration role, not the runtime app role.
