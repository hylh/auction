Fish Auction Implementation Roadmap

This roadmap starts from the current proof-of-concept: a runnable TanStack Start auction app with PostgreSQL, Drizzle schema and seed data, live auction detail, per-auction SSE, dashboard, inventory intake, admin statistics, metrics, simulator entrypoint, and initial unit tests.

  • Auction lifecycle
  • Transactional bidding
  • Admin controls
  • Integration coverage
  • Observability
  • Demo readiness

Current baseline

Area Implemented now Main remaining gap
Core app TanStack Start routes for /, /auctions/$auctionId, /inventory/new, /admin, and /metrics. Route-level error handling, loading states, and deeper UX polish.
Persistence PostgreSQL via Docker Compose, Drizzle schema, migration, deterministic seed data, and demo users. Lifecycle audit records, richer constraints, and integration test database setup.
Bidding Transactional bid writes, stale bid rejection, bid increment rules, seller protection, and typed bid errors. Server-level integration tests and stronger concurrency verification.
Real time Per-auction SSE broadcaster for accepted bids, rejected bids scoped to the actor, auction close, and sale completion events. SSE endpoint tests, reconnect behavior, and client-side resilience.
Admin and history Admin dashboard with auctions, completed sales, bid history, and aggregates. Withdrawal controls, status history, admin action history, and richer filtering.
Observability Structured logging helpers, OpenTelemetry span around bid placement, and Prometheus-style counters/gauges. Request-wide instrumentation, useful histograms, error counters, and docs.

Recommended implementation sequence

Phase 1 - Close the auction lifecycle gaps
  • Create auctions from listed fish inventory through a validated server function.
  • Move scheduled auctions to active when their start time arrives.
  • Close expired auctions automatically as sold or unsold.
  • Make close operations idempotent so repeated admin or scheduler calls are safe.
  • Broadcast auction.closed and sale.completed only after commit.
Phase 2 - Expand admin operations and auditability
  • Add admin withdrawal flows for fish items and active auctions.
  • Persist inventory status changes and admin actions in audit tables.
  • Show unsold auctions, withdrawn inventory, and status history on /admin.
  • Add filters by status, species, seller, buyer, and date range.
Phase 3 - Harden bidding and real-time behavior
  • Add integration tests for stale bid races against PostgreSQL row locks.
  • Test rejected bid SSE delivery only to the acting user.
  • Add EventSource reconnect guidance and cache reconciliation after reconnect.
  • Keep dashboard and admin aggregates invalidated rather than manually recomputed.
Phase 4 - Improve inventory and auction creation UX
  • Turn newly listed inventory into scheduled or active auctions from the UI.
  • Add validation feedback next to individual form fields.
  • Support optional image previews while preserving external URL validation.
  • Make money and kilogram inputs display parsed values before submit.
Phase 5 - Make observability production-minded
  • Wrap server functions and API handlers with request latency measurement.
  • Replace last-value duration gauges with useful buckets or summaries.
  • Add counters for validation failures, close failures, and simulator requests.
  • Document local metrics scraping and OpenTelemetry environment configuration.
Phase 6 - Upgrade the simulator and demo data
  • Let the simulator create auctions through the same application path as the UI.
  • Add options for auction count, bid interval, duration, buyer mix, and rejection rate.
  • Keep simulator data deterministic by default, with an optional random seed.
  • Exercise bidding, closing, metrics, logging, and SSE updates in one demo run.

Acceptance criteria by milestone

Milestone Done means
Lifecycle complete Listed fish can become auctions; scheduled auctions activate; expired auctions close; auctions with bids create sales; auctions without bids become unsold.
Admin audit ready Admins can close and withdraw records, and every admin action plus inventory status transition is visible in history.
Real-time confidence Multiple SSE listeners receive ordered accepted bid events; rejected events only reach the acting user; reconnects end in a consistent query cache.
Integration coverage Vitest covers server functions, PostgreSQL transactions, sale completion, unsold closing, metrics side effects, and broadcaster behavior.
Demo readiness A fresh clone can run pnpm install, docker compose up -d, pnpm db:migrate, pnpm db:seed, pnpm dev, and pnpm simulate to show live bid updates and completed sales.

Near-term backlog

  2. Add an auction lifecycle service that activates due auctions and closes expired active auctions.
  3. Add status history and admin action tables with Drizzle migration and seeded examples.
  4. Add PostgreSQL-backed Vitest integration tests for bid placement and auction closing.
  5. Add SSE endpoint tests for ordered accepted events and actor-scoped rejected events.
  6. Expand the simulator so it creates auctions, places bids, closes auctions, and reports metrics deltas.
  7. Improve form-level validation messages and admin filters before adding visual polish.

Guiding constraint

Keep the next work focused on correctness before polish: every lifecycle transition should be transactional, observable, covered by tests, and reflected in the dashboards without duplicating validation between client and server.
