Routing & Server Functions

How a filename becomes a URL, how a page differs from an API route, and how an imported function secretly becomes an HTTP call.

Why this lesson? TanStack Start & Router are on your "less familiar" list, and they hide a lot of magic. Two mechanisms explain almost everything you'll read in src/routes/: file-based routing (filename → URL) and server functions (typed RPC). Master these two and the whole client/server boundary becomes legible.

Mechanism 1 — the filename IS the URL

TanStack Router generates routes from filenames in src/routes/. Dots become slashes, $name is a path parameter, and an api. prefix is just convention for endpoints. The generated map lands in routeTree.gen.ts (never hand-edited).

FileURLKind
index.tsx/Page — dashboard
admin.tsx/adminPage
inventory.new.tsx/inventory/newPage
auctions.$auctionId.tsx/auctions/:auctionIdPage (dynamic)
api.auctions.$auctionId.events.ts/api/auctions/:auctionId/eventsAPI — SSE
api.simulate.ts/api/simulateAPI — POST
metrics.ts/metricsAPI — GET (HTML/text)

Read a dynamic param in a page with Route.useParams() — exactly what auctions.$auctionId.tsx does to get auctionId.

Mechanism 2 — page vs. API route

Both live in routes/; the difference is what the route exports.

Page exports a component

// index.tsx
export const Route = createFileRoute("/")({
  component: Dashboard,   // React renders
});

Renders React. Fetches data with TanStack Query calling a server function.

API exports server.handlers

// api.simulate.ts
export const Route = createFileRoute("/api/simulate")({
  server: { handlers: {
    POST: async ({ request }) => { ... }  // returns a Response
  }},
});

Raw HTTP. You read request and return a Response — no React.

Rule of thumb: component → it's a page; server.handlers → it's an endpoint. The app uses API routes only where it truly needs raw HTTP: the SSE stream, the simulator, and the /metrics output (which even returns Prometheus text/plain when asked).

Mechanism 3 — server functions: a typed RPC

This is the part that feels like magic. A page imports a function and calls it like local code — but it runs on the server. There's no fetch, no URL, no JSON wiring by hand.

// the page (browser): index.tsx
import { getDashboardFn } from "../server/functions";
const dashboard = useQuery({ queryKey: ["dashboard"], queryFn: () => getDashboardFn() });

// the definition (server): server/functions.ts
export const getDashboardFn = createServerFn({ method: "GET" })
  .handler(() => observeServerFn(() => getDashboardData()));

At build time TanStack Start splits this: the .handler body stays on the server, and the browser gets a stub that makes the HTTP request and deserializes the typed result. You get end-to-end types without writing or maintaining a REST contract.

The anatomy of every server function (all in server/functions.ts): createServerFn({ method }) → optional .inputValidator(...) → .handler(({ data }) => service(data)). Each one is a thin wrapper that delegates to a service in auction-commands/queries and wraps it in observeServerFn for metrics. tanstack start

Calling with input

// page passes { data }:
getAuctionDetailFn({ data: { auctionId } })
placeBidFn({ data: { auctionId, bidderId, amountCents, ... } })

The payload always goes under a data key. On the server, .inputValidator receives it (here it just passes unknown through), and the service function parses it with a Zod schema — so validation lives in the domain, not the boundary.

Putting it together: two ways data moves

DirectionMechanismExample
Browser → server (request/response)Server functionplaceBidFn, getDashboardFn
Server → browser (push)API route + SSEapi.auctions.$auctionId.events.ts

That's the same split you saw in Lesson 01: bids flow up through a server function; live updates flow down through the SSE endpoint. Routing and server functions are simply the two halves of that boundary.

Click an answer for instant feedback.

1. What URL does the file routes/inventory.new.tsx serve?

✓ Correct. Dots in the filename become slashes in the URL.
No — dots become slashes, so inventory.new.tsx → /inventory/new.

2. You open a file in routes/. How do you tell whether it's a page or an API endpoint?

✓ Right. It's about what the route exports — component (page) vs server.handlers (endpoint).
The extension is a hint at best. The real signal is the export: component = page, server.handlers = endpoint.

3. getDashboardFn() is imported into a browser component and called directly. What actually happens?

✓ Correct. That's the server-function magic: the handler body stays server-side; the client gets a typed RPC stub.
It's a server function — the handler executes on the server (DB and all) and the client receives the typed result. No SQL runs in the browser.

4. Why are the SSE stream, the simulator, and /metrics built as API routes instead of server functions?

✓ Right. Server functions are great for typed request/response, but raw Response control (streams, headers, Prometheus text) needs an API route.
It's about needing the raw Response: SSE streams, custom headers/redirects, and text/plain output don't fit the request/response server-function shape.
