The Map of src/

Place any file in the codebase into one of six layers — and know which layer to open before you even look.

Why this lesson? In Lesson 01 you followed one bid across files. That path crossed every layer of the app. Now we zoom out: if you can name the six layers and the one rule about how they're allowed to call each other, you can answer "where does this live?" for almost any feature — exactly the navigation skill your review role needs.

The whole src/ tree

src/ ├── routes/ # pages + API endpoints (file-based routing) ├── components/ # React UI pieces (admin/ dashboard cards) ├── server/ # business logic: commands, queries, transactions ├── domain/ # pure rules + shared types (no DB, no React) ├── db/ # Drizzle schema + Postgres client ├── theme/ # light/dark theming ├── scripts/ # seed + load simulator (run via pnpm/tsx) ├── styles.css # global CSS ├── router.tsx # builds the router + QueryClient └── routeTree.gen.ts # AUTO-GENERATED route tree — never edit

The one rule that ties it together

Dependencies flow inward: routes & components → server → domain & db.
The domain layer is the centre — it imports nothing from React, the database, or routes. That's why bid rules, money math, and event types are trivially unit-testable. If you ever see domain/ importing from server/ or routes/, something is upside-down.

The six layers in detail

1 · Routes src/routes/

Both pages and API endpoints. The filename is the URL (TanStack Router file-based routing). Dots become slashes; $param is a path variable.

  • __root.tsx — the shell wrapping every page: QueryClientProvider, <Outlet/>, theme, <head>.
  • index.tsx → / live dashboard · admin.tsx → /admin · inventory.new.tsx → /inventory/new
  • auctions.$auctionId.tsx → /auctions/:id (the page from Lesson 01).
  • api.auctions.$auctionId.events.ts → the SSE stream · api.simulate.ts · metrics.ts → the /metrics endpoint.
Pages vs APIs share this folder: a file exporting a component is a page; one whose route declares server.handlers (like the api.* files) is a raw HTTP endpoint. routeTree.gen.ts is generated from these — never hand-edit it.

2 · Components src/components/

Reusable React UI. Currently focused on the admin dashboard, which is broken into one card per concern.

  • admin/ — auctions-card, bid-history-card, completed-sales-card, summary-cards, status-history-card, filters, tabs…
  • admin/use-admin-mutations.ts — a hook bundling the admin write actions (close, withdraw) — UI calling into server functions.
  • admin/types.ts — view-model types shared across the admin cards.

3 · Server src/server/

All business logic. Split by responsibility — and crucially split between writes (commands) and reads (queries).

  • functions.ts — the only client↔server bridge: every createServerFn lives here and delegates to a service.
  • auction-commands.ts — writes: createFishItem, createAuction, placeBid, closeAuction, withdraw… (transactional).
  • auction-queries.ts / auction-admin-queries.ts — reads: dashboard, auction detail, admin/statistics.
  • auction-lifecycle.ts · auction-clock.ts — closing auctions & auto-advancing expired ones.
  • row-locks.ts (the FOR UPDATE helpers) · bid-selection.ts · auction-audit.ts (admin-action & status-change logging) · auction-mappers.ts (row → view model).
  • auction-service.ts — a barrel file that just re-exports the above (the public surface other code imports).
Reads and writes are separated on purpose. When reviewing a mutation, open auction-commands.ts; for a screen's data, open the matching *-queries.ts.

4 · Domain src/domain/

The pure heart: business rules, value conversions, shared types, validation. No database, no React. This is where the .test.ts files cluster.

  • bid-rules.ts — evaluateBid(), the accept/reject decision & typed error codes.
  • events.ts — auction event types + the in-process event bus (Lesson 01, step 6).
  • validation.ts — the Zod schemas shared by server and UI (one source of truth for input shape).
  • money.ts (cents math) · weight.ts (grams↔kg) — never floating-point money.
  • constants.ts — the enums: species, statuses, roles, currency (NOK), demo-user IDs.
  • metrics.ts · metric-exposition.ts · logger.ts — observability primitives.

5 · Database src/db/

The Drizzle layer — schema is the source of truth that migrations are generated from.

  • schema.ts — tables (users, fish_items, auctions, bids, sales, plus rejected_bids, audit tables), enums, indexes, and relations.
  • client.ts — the Postgres connection pool + Drizzle client (db), tagged application_name=auction-app for the metrics page.
Enums in schema.ts are built from domain/constants.ts — so the database and the app literally can't disagree on valid species/statuses. unfamiliar stack

6 · Infra & tooling theme/ · scripts/ · router.tsx

Everything supporting the app but not part of the bid path.

  • router.tsx — builds the router, creates the per-request QueryClient, wires defaultPreload: "intent".
  • theme/ — cookie-backed light/dark theme (server reads the cookie, a tiny inline script avoids flash).
  • scripts/seed.ts (→ pnpm db:seed) · factories.ts · seed-data/ — deterministic demo data & scenarios.
  • scripts/simulate.ts (→ pnpm simulate) — the load generator.

Check yourself 2 min

Where does it live? Click an answer for instant feedback.

1. You want to add a new fish species, "plaice", so it's valid everywhere — DB enum, validation, and UI dropdown. Where's the single source you edit first?

✓ Correct. FISH_SPECIES in constants.ts feeds the DB enum, Zod, and the UI — change it there and regenerate the migration.
Not the best entry point. schema.ts derives its enum from domain/constants.ts, and the UI reads it too. Edit constants.ts first.

2. A reviewer asks "what query powers the admin statistics page?" Which folder do you open?

✓ Right. Read models live in server/*-queries.ts; the admin cards in components/ just render what those queries return.
Close, but components/admin/ only renders the data. The actual aggregate SQL is in server/auction-admin-queries.ts.

3. Which statement matches the dependency rule?

✓ Exactly. The domain is the pure centre; routes/components → server → domain/db, never the reverse.
No. Dependencies flow inward: the domain is pure and imports nothing outward, while routes do call server functions.

4. You see routeTree.gen.ts in a diff. What should you do?

✓ Correct. It's auto-generated from the files in routes/. You edit route files; this regenerates.
It's machine-generated from routes/. Don't edit it by hand or move it — just let route-file changes regenerate it.
