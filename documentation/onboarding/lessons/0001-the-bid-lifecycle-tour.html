The Life of a Bid

Follow one bid from the buyer's click all the way to every other browser updating — through the real files in this repo.

Why this lesson? Your mission is to navigate and review the code you built with AI. Almost every important pattern in this codebase shows up on the path a single bid travels: file-based routing, a TanStack Start server function, Zod validation, a transactional Drizzle write, pure domain rules, an in-process event bus, an SSE stream, and a TanStack Query cache patch. Learn this one path and you can find your way to anything.

The whole journey on one line

auctions.$auctionId.tsx→ placeBidFn→ placeBid()→ Zod→ DB tx + lock→ evaluateBid()→ insert bid→ event bus→ SSE→ setQueryData

That is the spine of the app. Now let's walk each vertebra with the file and line to open.

1

The buyer submits — a React component calls a "server function"

src/routes/auctions.$auctionId.tsx · the placeBid mutation

The auction page is an ordinary TanStack Query screen. Submitting the form runs a useMutation whose mutationFn calls placeBidFn(...). It looks like a local function call, but placeBidFn is a server function: TanStack Start turns it into an HTTP request to the server. You never write fetch or a REST route for it.

// imported from the server, called from the browser
import { placeBidFn } from "../server/functions";

const placeBid = useMutation({
  mutationFn: () => placeBidFn({ data: { auctionId, bidderId, amountCents, ... } }),
});
Mental model: a server function = a typed RPC. The client imports it; TanStack Start runs it on the server and ships back the result, fully typed.
2

The server boundary — validate the request shape

src/server/functions.ts · placeBidFn

export const placeBidFn = createServerFn({ method: "POST" })
  .inputValidator((data: unknown) => data)
  .handler(({ data }) => observeServerFn(() => placeBid(data)));

This file is the only bridge between browser and business logic. Every server function is thin: declare the HTTP method, accept input, hand off to a service function, and wrap it in observeServerFn (which records metrics and tags Zod failures). Note it delegates to placeBid() — the real work lives elsewhere.

3

Parse with Zod, then open a transaction with a row lock

src/server/auction-commands.ts · placeBid()

const data = bidInputSchema.parse(input);   // Zod: untrusted -> typed
await advanceAuctionLifecycle();            // auto-close expired auctions first

return await db.transaction(async (tx) => {
  await lockAuctionRow(tx, data.auctionId); // SELECT ... FOR UPDATE
  const auction = await tx.query.auctions.findFirst({ ... });
  const currentHighestBid = await findHighestBidWithBidder(tx, data.auctionId);
  ...
The key concurrency move: lockAuctionRow issues a Postgres SELECT … FOR UPDATE on that one auction. Two simultaneous bids on the same auction are forced to take turns, so the "current highest bid" can't be read stale. This is how the app stays correct under load. unfamiliar stack
4

Decide accept-or-reject with a pure function

src/domain/bid-rules.ts · evaluateBid()

The command gathers facts (amount, current highest, starting price, minimum increment, auction window, seller vs. bidder) and passes them to evaluateBid() — a pure function with no database access. It returns either { ok: true } or a typed error code: stale bid, auction not active, seller bidding on own auction, insufficient increment, or invalid amount.

const bidDecision = evaluateBid({ amountCents, currentHighestBidCents, ... });

if (!bidDecision.ok) {
  await tx.insert(rejectedBids).values({ ...bidDecision }); // audited, not swallowed
  return { ok: false, code: bidDecision.code, message: bidDecision.message, ... };
}
Why pure? It makes the auction rules trivially unit-testable (see src/domain/bid-rules.test.ts) without a database. Rejected bids are persisted, never silently dropped.
5

Accepted: insert the bid, build an event — still inside the transaction

src/server/auction-commands.ts · end of the tx block

const [insertedBid] = await tx.insert(bids).values({ auctionId, bidderId, amountCents }).returning();
return { ok: true, event: bidAcceptedEvent(data.auctionId, snapshot) };

The bid row and the event object are created inside the transaction, but the event is only published after the transaction commits (next step). That ordering guarantees nobody hears about a bid that might roll back.

6

Broadcast — publish to the in-process event bus

src/domain/events.ts · publishAuctionEvent()

// back outside the transaction, after commit:
if (result.ok) {
  publishAuctionEvent(result.event);   // notifies every SSE subscriber for this auction
}

The bus is a small in-memory Map<auctionId, Set<subscriber>>. One subtlety worth noticing: a bid.rejected event is only delivered to the user who made it (see shouldDeliverAuctionEventToSubscriber) — others don't see your failures.

7

Stream it out over Server-Sent Events

src/routes/api.auctions.$auctionId.events.ts

This is a plain API route (not a page). It opens a ReadableStream with Content-Type: text/event-stream, subscribes to the bus for that auction, and writes each event in SSE wire format. When the browser disconnects, request.signal fires abort and it unsubscribes.

controller.enqueue(encoder.encode(`event: ${eventName}\ndata: ${JSON.stringify(data)}\n\n`));
SSE vs WebSocket: SSE is one-way (server→browser), plain HTTP, and auto-reconnects. Perfect here because bids flow up via the server function and updates flow down via the stream. unfamiliar stack
8

Every browser patches its cache — no full refetch

src/routes/auctions.$auctionId.tsx · the EventSource effect

const eventSource = new EventSource(`/api/auctions/${auctionId}/events?userId=${bidderId}`);

// on a bid.accepted message:
queryClient.setQueryData(["auction", auctionId], (current) => ({
  ...current,
  currentHighestBid: parsed.currentHighestBid,
  bids: [parsed.bid, ...current.bids],     // prepend, dedupe by bidId
}));
queryClient.invalidateQueries({ queryKey: ["dashboard"] }); // aggregates: refetch

Two strategies, used deliberately: the detail view is patched directly (cheap, instant), while aggregate screens (dashboard, admin stats) are invalidated so the server recomputes them. That's the loop closed — your bid is now on everyone's screen.

2. Why does placeBid() call lockAuctionRow(tx, …) before reading the highest bid?

✓ Exactly. SELECT … FOR UPDATE forces simultaneous bidders to take turns, which is what prevents stale-bid races.
No — it's about concurrency. The lock (SELECT … FOR UPDATE) serializes simultaneous bids so the "current highest" read is never stale.

3. A bid is accepted. In what order do these happen?

✓ Right. The event is published only after commit, so nobody is told about a bid that could still roll back.
Look again at step 5–6: the event is published after the transaction commits, never before — otherwise a rolled-back bid could be broadcast.

4. The detail page is patched with setQueryData, but the dashboard is invalidated. Why the difference?

✓ Yes. Patch the one thing you know changed; let the server recompute totals/averages rather than re-deriving every aggregate client-side.
Re-read step 8: the dashboard does use Query. The point is that aggregates (totals, averages) are cheaper to recompute server-side via invalidation than to patch by hand.
