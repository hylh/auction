The Database Schema

Read db/schema.ts like a native: why money is an integer, why nothing is deleted, and how indexes quietly power the stats page.

Why this lesson? Drizzle & SQL are on your "less familiar" list, yet the schema is the spine everything else hangs off. Once you can read these eight tables and the four design decisions baked into them, the server queries from Lesson 02 stop being mysterious — they're just walks across this graph. One file: src/db/schema.ts.

The eight tables at a glance

users (seller / buyer / admin) │ │ owns ▼ fish_items ──(1 fish, many)──▶ auctions ──(1 auction, many)──▶ bids │ │ │ │ │ │ winning bid │ ▼ ▼ └────────────────────────────▶ sales ◀──────────────────────┘ audit / history (write-only): rejected_bids · every bid that was turned away (with code + reason) inventory_status_changes · every fish status transition admin_actions · every close / withdraw / create an admin did

Five core tables model the auction; three audit tables record history. Defined with Drizzle's pgTable(...), each followed by relations(...) that let you traverse with tx.query.auctions.findFirst({ with: { fishItem: { with: { seller: true }}}}) — exactly the nested read you saw in placeBid().

Anatomy of one table

Here's fish_items, lightly trimmed — every pattern in the schema appears here:

export const fishItems = pgTable("fish_items", {
  id: uuid("id").primaryKey().defaultRandom(),
  species: fishSpeciesEnum("species").notNull(),        // enum from domain/constants.ts
  displayName: text("display_name").notNull(),
  weightGrams: integer("weight_grams").notNull(),       // kg shown in UI, GRAMS stored
  startingPriceCents: integer("starting_price_cents").notNull(), // money = integer CENTS
  currency: text("currency").notNull().default("NOK"),
  sellerId: uuid("seller_id").notNull().references(() => users.id), // FK
  status: inventoryStatusEnum("status").notNull().default("draft"),
  ...timestampsWithUpdate(),                            // createdAt + updatedAt
}, (table) => [
  index("fish_items_seller_idx").on(table.sellerId),    // indexes declared inline
  index("fish_items_species_idx").on(table.species),
  index("fish_items_status_idx").on(table.status),
]);
How to read a column: type("db_name") then a chain of modifiers — .notNull(), .default(...), .references(() => …) (foreign key), .primaryKey(). The second argument to pgTable is an array of indexes. That's the whole grammar. drizzle

The four decisions baked into the schema

1 · Money is integer cents; weight is integer grams

You will never see a float for money. Prices, bids, and sale amounts are integer … _cents; fish weight is weight_grams. The UI converts at the edges via domain/money.ts and domain/weight.ts. This avoids floating-point rounding errors on currency.

2 · Enums are shared with the app, not redefined

export const fishSpeciesEnum = pgEnum("fish_species", FISH_SPECIES);   // FISH_SPECIES from domain/constants.ts
export const inventoryStatusEnum = pgEnum("inventory_status", INVENTORY_STATUSES);

The Postgres enums are built from the same TypeScript constants the validation and UI use — so the database physically cannot store a status the app doesn't recognise.

3 · History is never deleted — it's appended

A rejected bid is written to rejected_bids (with its code and reason). Withdrawing an auction flips a status and writes an inventory_status_changes row plus an admin_actions row. Nothing is hard-deleted — the MVP preserves a full audit trail for admin review and statistics.

4 · A sale can only happen once — enforced by the database

uniqueIndex("sales_auction_unique_idx").on(table.auctionId),      // one sale per auction
uniqueIndex("sales_winning_bid_unique_idx").on(table.winningBidId), // one sale per winning bid
These unique indexes are a correctness guarantee, not just speed: even with a bug or a race, Postgres refuses to record two sales for the same auction.

How indexes power the stats page

The admin statistics aren't a separate analytics store — they're live aggregate queries over these tables, made fast by the indexes above. Straight from server/auction-admin-queries.ts:

sql`coalesce(sum(${sales.amountCents}), 0)`        // total sales
sql`coalesce(round(avg(${bids.amountCents})), 0)`  // average bid price

db.select({ species: fishItems.species, bidCount: sql`count(*)` })
  .from(bids).innerJoin(auctions, ...).innerJoin(fishItems, ...)
  .groupBy(fishItems.species);                      // most popular species

"Most popular fish" = bids grouped by species. "Total sales" = sum over sales. Each is one SQL statement leaning on a matching index (bids_auction_amount_idx, sales_completed_at_idx, …).

Where migrations come from

You don't write SQL DDL by hand. Edit schema.ts, then pnpm db:generate diffs it and writes a migration into drizzle/; pnpm db:migrate applies it. The schema file is the source of truth. drizzle kit
