Fish Auction Code Review and Architecture Improvements

This review looks for deepening opportunities in the current proof-of-concept: places where a Module can hide more behaviour behind a smaller Interface, improve Locality, and give callers and tests more Leverage.

  • Auction lifecycle
  • Transactional bidding
  • Realtime bid chain
  • Admin statistics
  • Validation
  • Observability
  • Simulator

Executive summary

The project has a strong vertical slice: PostgreSQL persistence, Drizzle schema, transactional bid writes, SSE updates, admin dashboards, metrics, simulator entry points, and Vitest coverage for important domain rules. The main improvement opportunity is not more scaffolding; it is increasing Depth at the seams that already exist.

The highest-leverage work is to deepen the auction lifecycle, admin query/statistics, validation, realtime event, metrics, and simulator Modules. These changes would concentrate domain invariants and persistence details in fewer places, make tests exercise behaviour through stable Interfaces, and reduce the amount of code a maintainer must read to change a fish auction rule safely.

Context note: no CONTEXT.md or ADR folder was present. This review uses documentation\auction-project-decisions.html, the project instructions, and the existing source as the domain record. No ADR conflicts were found.

Prioritized improvement map

Priority Deepening opportunity Why now
1 Auction lifecycle and bidding Module Core business behaviour; every future feature will touch it.
2 Admin query and statistics Module Current filtering and aggregate computation leave PostgreSQL leverage unused.
3 Shared validation and domain vocabulary Module Duplicated status, species, money, and weight rules can drift.
4 Realtime auction event Module SSE works, but ordering, replay, and cache projection need one clearer seam.
5 Metrics exposition Module Prometheus text and visual metrics rendering are duplicated and coupled.
6 Simulator command Module Load generation should have a narrower contract and clearer pacing ownership.

Findings

1. Deepen the auction lifecycle and bidding Module

Files: src\server\auction-service.ts, src\domain\bid-rules.ts, src\domain\events.ts, src\server\functions.ts, src\routes\auctions.$auctionId.tsx.

Problem: auction-service.ts has become the main Module for dashboard reads, auction detail reads, fish inventory creation, auction creation, bid placement, manual close, automatic lifecycle advancement, withdrawal, admin history, filtering, row mapping, event publication, metrics, and audit records. The Interface is a list of many exported functions, but callers still need to understand transaction ordering, status transitions, event timing, metrics side effects, and audit side effects. That is a shallow Interface around a large Implementation.

Deletion test: deleting the current file would scatter transaction locks, inventory status transitions, sale creation, event publishing, and metrics increments across routes and scripts. That means the Module is earning its keep, but its Interface has grown too wide for the amount of behaviour it hides.

Solution: deepen around the domain seams that already exist: bidding, auction lifecycle, fish inventory, and read models. Keep bid acceptance and auction close as behaviour-rich entry points that own transaction ordering, committed-state reads, audit writes, metrics, and event publication. Move row shaping and admin read models behind a separate query Module so command callers do not inherit read-model complexity.

Benefits: Locality improves because a bid rule or lifecycle transition can change in one Module instead of across exported functions and routes. Leverage improves because tests can cross the same Interface as callers: "place a bid", "close an auction", and "withdraw inventory" should verify persisted rows, emitted events, metrics, and audit records without testing private helpers.

2. Push admin filtering and statistics into a PostgreSQL-backed query Module

Files: src\server\auction-service.ts, src\routes\admin.tsx, src\db\schema.ts.

Problem: admin data is loaded broadly, filtered in memory, then sliced or aggregated in TypeScript. Examples include loadAuctionSummaries, loadLatestBids, loadRecentSales, and the popular fish calculation in getAdminData. This contradicts the project decision that statistics should be computed with live PostgreSQL aggregate queries and appropriate indexes. It also couples the admin route to arrays that may become expensive as bid history grows.

Deletion test: deleting the filter helpers would force equivalent status, species, seller, buyer, and date rules to reappear in every admin query. They are real behaviour, but they live behind the wrong seam: after data has already crossed from PostgreSQL into memory.

Solution: create a deep admin query Module that owns the filter Interface and translates it to Drizzle conditions and aggregate SQL. It should return exactly the dashboard read models the UI needs: completed sales, unsold auctions, bid history, status history, admin actions, total sales, average bid price, and popular fish with bid count, kilograms sold, and sale value.

Benefits: PostgreSQL becomes the Adapter doing set operations and aggregation. Locality improves because filter semantics live in one Module. Leverage improves because one integration test at the query Interface can cover all surfaces that currently depend on repeated in-memory filtering.

3. Consolidate validation, status vocabulary, money, and weight rules

Files: src\domain\validation.ts, src\domain\constants.ts, src\domain\money.ts, src\domain\weight.ts, src\db\schema.ts, src\routes\inventory.new.tsx, src\routes\admin.tsx.

Problem: species, inventory statuses, auction statuses, currency, money parsing, and weight parsing are spread across several Modules. validation.ts mixes fish input, bid input, auction timing, close/withdraw commands, simulator input, and admin filters. schema.ts repeats enum vocabularies that also exist in domain constants and route-local arrays. The inventory route validates with Zod, then validates money and weight again with conversion helpers, and uses a dummy fish id to satisfy an auction schema before the real fish exists.

Solution: make the domain vocabulary a deeper Module: canonical species, inventory status, auction status, currency, money input, and weight input should be exported from one place and reused by Zod schemas, Drizzle enum declarations, route options, seed data, and tests. Keep route form code as an Adapter that maps browser strings into shared input shapes rather than re-implementing validation decisions.

Benefits: Locality improves because changing a fish species, status, or unit invariant happens once. Leverage improves because tests for money and weight parsing protect both server functions and UI form validation. The Interface becomes smaller: callers ask the domain Module to parse fish, auction, bid, or admin input instead of knowing which combination of Zod schemas and conversion helpers to call.

4. Deepen the realtime auction event Module

Files: src\domain\events.ts, src\routes\api.auctions.$auctionId.events.ts, src\server\auction-service.ts, src\routes\auctions.$auctionId.tsx, src\domain\events.test.ts, src\server\auction-service.integration.test.ts.

Problem: the in-memory broadcaster is small and useful, but the complete realtime Interface spans several places: event type definitions, subscriber filtering, SSE formatting, connected messages, post-commit publication from bid and close operations, and TanStack Query cache patching in the auction detail route. The current test covers the in-memory broadcaster but not the full "accepted bid commits, then all listeners receive the ordered event" behaviour required by the project brief.

Solution: deepen an auction event Module that owns event construction, subscriber visibility, SSE encoding, and delivery guarantees. Keep the transport-specific stream as an Adapter at the HTTP seam. Add a server-level integration test that subscribes listeners, places a bid through the same command Interface as the application, and asserts that ordered accepted events arrive only after persistence succeeds. Add a rejected-bid test that verifies the acting-user-only rule across the SSE seam.

Benefits: Locality improves because realtime invariants are no longer split between domain events, route stream code, server command code, and client cache updates. Leverage improves because one event Interface can support future replay, heartbeat, or reconnect behaviour without forcing every route to learn those details.

5. Separate metrics collection, Prometheus exposition, and visual rendering

Files: src\domain\metrics.ts, src\server\database-metrics.ts, src\routes\metrics.ts, src\domain\metrics.test.ts.

Problem: the metrics code works, but two Modules hand-roll Prometheus text formatting and both define number-formatting helpers. The visual metrics page is also a large route-local string renderer coupled to both in-memory counters and database metrics. The Interface to metrics is therefore broader than it needs to be: callers must know which metrics source returns text, which returns a snapshot, and how route rendering combines them.

Deletion test: deleting either text formatter would cause similar HELP, TYPE, label, bucket, and number-formatting logic to reappear. That is a sign the formatting behaviour belongs behind a shared exposition seam.

Solution: create a metrics exposition Module that accepts metric families and renders Prometheus text. Treat in-memory domain metrics and PostgreSQL database metrics as Adapters that produce metric families. Move the visual page rendering behind a presenter Module or keep it route-local only after the data shape has been normalized.

Benefits: Locality improves because Prometheus escaping, label formatting, histograms, counters, and gauges are verified in one place. Leverage improves because adding a new request, bid, or database metric does not require copying exposition details.

6. Narrow and clarify the simulator command Interface

Files: src\server\simulator-service.ts, src\routes\api.simulate.ts, src\scripts\simulate.ts, src\domain\validation.ts, src\server\auction-service.integration.test.ts.

Problem: the simulator deliberately exercises the real service path, which is a good decision. However, the server-side Interface still exposes scheduling-looking fields such as intervalMs even though pacing happens in the CLI script. The server Implementation also directly orchestrates fish creation, auction creation, bid placement, dashboard fallback, auction detail loading, and close operations. This makes the simulator useful but broad.

Solution: split the simulator into a CLI pacing Module and a server "single simulation step" Module. The server step should take a narrow request, such as create one auction, place one bid against supplied auction ids, or close supplied auctions. The CLI can own intervals, run duration, and load profile composition. Keep the auction command Module as the Adapter used by the simulator so validation, metrics, and events still exercise the production path.

Benefits: Locality improves because load pacing and domain mutation semantics are separate. Leverage improves because the same server step can be tested deterministically and reused by scripts, demo buttons, or future load profiles without broadening the auction command Interface.

7. Move route-local orchestration into feature Modules where behaviour repeats

Files: src\routes\auctions.$auctionId.tsx, src\routes\inventory.new.tsx, src\routes\admin.tsx, src\routes\index.tsx.

Problem: routes currently own UI rendering plus query keys, mutation invalidation, cache patching, form validation mapping, filter state, and status-specific action rules. Some of this is normal for a POC, but as the product grows these route Modules will become shallow: the Interface is just a route, while the Implementation contains many behaviours that need independent tests.

Solution: introduce feature Modules only where repeated behaviour appears: live auction detail state, inventory intake form state, and admin filters/actions. Do not extract presentation for its own sake. Use the deletion test: if deleting a feature Module would force cache-update, validation, or action-rule complexity back into several routes, it is worth keeping.

Benefits: Locality improves because query invalidation and route-specific domain decisions are not mixed with markup. Leverage improves because tests can exercise form parsing, cache projection, and action enablement through a feature Interface while the route stays a thin Adapter to TanStack Router.

8. Expand tests at the behavioural Interfaces, not around private helpers

Files: src\domain\bid-rules.test.ts, src\domain\events.test.ts, src\domain\money-weight.test.ts, src\domain\metrics.test.ts, src\server\auction-service.integration.test.ts.

Problem: unit tests cover bid rules, events, metrics, money, and weight; integration tests cover stale bid races and simulator request timing. The important gaps are at the behavioural Interfaces: seller cannot bid on own auction through persistence, non-admin users cannot create/close/withdraw auctions, accepted bids publish after commit, rejected bids are actor-only, auction close creates the expected sale and inventory status records, and unsold close returns inventory to listed.

Solution: add integration tests through the deepened command and event Interfaces. Replace tests that know private mapping/filter helpers with tests that assert observable results: rows, typed errors, events, metrics counters, and dashboard/admin read models.

Benefits: The Interface becomes the test surface. Tests survive internal refactors because they describe fish auction behaviour rather than helper structure.

Recommended first implementation slice

  1. Deepen the auction command Module around bid placement and auction close first. Keep the public behaviour the same, but make the command Interface own transaction, audit, event, and metrics side effects.
  2. Add integration tests for seller-own-auction rejection, admin permission failures, accepted bid event delivery after commit, sale completion, and unsold close.
  3. Move admin statistics to PostgreSQL aggregate queries once the command behaviour is pinned down by tests.
  4. Consolidate domain vocabulary and validation before adding new species, statuses, or dashboard filters.

This sequence protects the core fish auction lifecycle first, then improves query performance and maintainability without changing user-visible routes.
