1.4 Seeded Bids

The default scenario seeds ~32 accepted bids across its auctions using deterministic multi-round escalation that alternates buyers and ascends by the configured minimum increment:

AuctionBidsBuyersNotes
Salmon (active)12Oslo / Bergen alternatingEscalates from the 1 800 NOK starting price.
Cod (active)10Bergen / Oslo alternatingEscalates from the 1 400 NOK starting price.
Tuna (closed)6Oslo / Bergen alternatingHighest bid becomes the completed sale.
Lifecycle halibut (closed)4Oslo / Bergen alternatingHighest bid becomes the completed sale.

Bids are inserted in chronological order, so the last bid on each auction is always the current highest. Closed-auction sales reference the actual highest seeded bid (id, amount, and buyer), keeping referential integrity intact.

1.5 Completed Sales

The default scenario records two completed sales, each tied to the highest bid on a closed auction: the bluefin tuna and the lifecycle demo halibut. Both sales reference the real winning bid id, amount, and buyer.

1.6 Additional Seeded Records

  • Inventory status changes – 10 records, including the full draft → listed → in_auction → sold lifecycle and a null → draft entry for the unfinished draft lot.
  • Admin actions – 6 records: auction creations, auction closes, and an inventory withdrawal.

1.7 Seed Scenario Profiles

The seed script accepts a named scenario via --scenario <name> or the SEED_SCENARIO environment variable. Each scenario deletes existing data, reseeds the demo users, then builds its dataset:

2. How Bids Are Created

2.1 Entry Points

  1. UI form – The auction detail page calls the placeBidFn TanStack Start server function.
  2. Simulator – pnpm simulate drives runSimulation() which calls placeBid() programmatically.

2.2 Bid Input Schema

{
  auctionId: string (UUID),      // Target auction
  bidderId: string (UUID),       // The buyer placing the bid
  amountCents: number,           // Bid amount in integer cents (NOK øre)
  expectedHighestBidCents: number | null  // Optimistic concurrency token
}

2.3 Bid Validation Rules (bid-rules.ts)

The evaluateBid() function enforces these rules in order:

  1. INVALID_AMOUNT – Amount must be a positive integer.
  2. AUCTION_NOT_ACTIVE – Auction status must be "active" and current time must be within [startsAt, endsAt).
  3. SELLER_OWN_AUCTION – Sellers cannot bid on their own fish.
  4. STALE_BID – expectedHighestBidCents must match the committed highest bid (optimistic concurrency check).
  5. INSUFFICIENT_INCREMENT – Bid must be ≥ starting price (if first bid) or ≥ current highest + minimum increment.

2.4 Transaction Flow

  1. Zod validates the input shape.
  2. advanceAuctionLifecycle() expires any timed-out auctions.
  3. A PostgreSQL transaction starts with a SELECT ... FOR UPDATE row lock on the auction.
  4. The auction and its current highest bid are loaded.
  5. evaluateBid() applies business rules.
  6. On success: bid is inserted, a BidSnapshot is assembled.
  7. After commit: SSE event (bid.accepted or bid.rejected) is published, metrics are recorded.

2.5 Simulator Bid Creation

The simulator (pnpm simulate) creates bids by:

  • Optionally creating new fish + auctions first.
  • Computing a valid next bid amount with the shared buildValidBidInput() factory (see §3.5).
  • Applying a configurable rejectionRate (default 25%) to intentionally submit below-floor bids for testing rejection paths.
  • Calling the same placeBid() service function used by the UI.
  • Optionally closing created auctions and generating sales.

Multi-round escalation. Passing bidRounds > 0 switches the simulator into a competitive mode: for each round it walks every targeted auction and alternates buyers, so prices climb gradually across rounds. This exercises the increment and concurrency logic far more thoroughly than a single flat bid loop.

Multi-auction targeting. The auctionIds input now accepts several existing auctions (previously capped at one), so the simulator can drive realistic activity across all currently active seeded auctions without creating new ones.

3. Implemented Improvements in Data Creation

All of the improvements below have been implemented. Each block describes the original gap and how it was resolved.

3.1 Richer Bid Seeding

Resolved. The default seed now creates roughly 32 accepted bids across four auctions using deterministic multi-round escalation (12 salmon, 10 cod, 6 tuna, 4 lifecycle halibut), instead of the previous 4 bids. This makes the default state immediately useful for dashboard and statistics work without running the simulator.

3.2 Simulator Multi-Round Bid Escalation

Resolved. The simulator accepts a bidRounds option. When set, it runs that many rounds across every targeted auction, alternating buyers each round so prices escalate gradually — modelling realistic competition and exercising the concurrency/increment logic. From the CLI, --bid-rounds N switches into a single batch request that drives this mode.

3.3 Configurable Seed Scenario Profiles

Resolved. The seed script supports named scenarios (default, busy-auction, all-expired, no-bids) via --scenario or SEED_SCENARIO. See §1.7. Developers and testers can reproduce specific UI states and edge cases without editing the seed script.

3.4 Submission-Ready Bid Context Helper

Resolved. getBidSubmissionContext(auctionId) (exported from auction-service) atomically reads the current highest bid and returns a payload ready for bid submission: auctionId, auctionStatus, startingPriceCents, minimumIncrementCents, currentHighestBid, expectedHighestBidCents, and the precomputed nextMinimumBidCents. Scripts, tests, and external integrations no longer have to compute the optimistic-concurrency token by hand.

3.5 Shared Bid Factory Functions

Resolved. src/domain/bid-builder.ts provides pure helpers nextMinimumBidCents() and buildValidBidInput(). The simulator and the bid-rules floor calculation both use them, eliminating duplicated, drift-prone logic. The builder computes the next valid amount (optionally raised by incrementSteps) and returns a ready-to-submit payload.

3.6 Full Inventory Lifecycle in the Seed

Resolved. The default seed now includes a draft fish item and a "Lifecycle demo halibut" that walks the complete draft → listed → in_auction → sold path with matching inventory status changes and admin actions. Every inventory status is represented end-to-end.

3.7 Simulator Targets Multiple Existing Auctions

Resolved. The auctionIds input is no longer capped at one auction, and the CLI parses --auction-ids (comma or space separated). Supplying existing auction ids defaults the created-auction count to zero, so the simulator drives activity across all targeted auctions without creating new inventory.

4. Quick Reference: Creating a Bid

Via the UI

  1. Navigate to an active auction detail page (/auctions/$auctionId).
  2. Select a buyer from the user switcher.
  3. Enter a bid amount ≥ current highest + minimum increment.
  4. Submit. The bid is validated server-side and an SSE event confirms acceptance or rejection.

Via the Simulator CLI

# Create 1 new auction and place 1 bid on it, then close it:
pnpm simulate

# Multi-round competition across the two seeded active auctions (no new inventory):
pnpm simulate --auction-ids 10000000-0000-4000-8000-000000000001,10000000-0000-4000-8000-000000000002 --bid-rounds 4

# Place a few bids on a single existing auction without creating new ones:
pnpm simulate --auction-ids 10000000-0000-4000-8000-000000000002 --bid-count 3

Supplying --auction-ids defaults the created-auction count to zero. Add --close to close the targeted auctions when finished.

Via Direct Server Function (tests/scripts)

import { getBidSubmissionContext, placeBid } from "./server/auction-service";

// Atomically fetch the submission-ready context (no manual token math):
const context = await getBidSubmissionContext("10000000-0000-4000-8000-000000000001");

const result = await placeBid({
  auctionId: context.auctionId,
  bidderId: "00000000-0000-4000-8000-000000000003",  // Oslo buyer
  amountCents: context.nextMinimumBidCents,
  expectedHighestBidCents: context.expectedHighestBidCents,
});

if (result.ok) {
  console.log("Bid accepted:", result.event.bid);
} else {
  console.log("Bid rejected:", result.code, result.message);
}
