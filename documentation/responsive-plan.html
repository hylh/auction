Full-Responsiveness Plan

A review of the current mobile-layout implementation plus a phased plan to make the entire site responsive — not just reflow at phone width, but scale fluidly across phone, tablet, and desktop. The headline gap today is typography: almost every font size is a fixed rem value, so text never adapts between breakpoints.

What was just fixed

FIXED  Admin filters invisible on desktop. The admin filter panel uses a <details> element that is collapsible on phones and meant to be always-open on desktop. The desktop rule relied on .admin-filters-details > :not(summary) { display: block } to override the old browser behaviour of hiding closed-details children with display: none.

Modern Chromium (and the spec) no longer hide that content with display. Instead the children are wrapped in a ::details-content pseudo-element that is hidden via content-visibility: hidden while closed. The old display override therefore had no effect and the filters vanished on desktop. The fix forces the pseudo-element visible at the desktop breakpoint, keeping the legacy display override for engines that don't yet implement ::details-content: 

@media (min-width: 641px) {
  .admin-filters-details > summary { display: none; }

  /* Newer engines: closed content hidden via ::details-content */
  .admin-filters-details::details-content { content-visibility: visible; }

  /* Older engines: closed children hidden via display:none */
  .admin-filters-details > :not(summary) { display: block; }
}

Follow-up worth considering: the always-open-desktop / collapsible-mobile pattern built on a single <details> is inherently fragile (it fights the element's own state machine). A cleaner long-term approach is a small React component that renders a plain always-visible panel on desktop and a real toggle on mobile, driven by a media query hook — removing the CSS override entirely. Captured in Phase 3 below.

Review of the mobile-layout implementation

The existing mobile work (see documentation/mobile-plan.html) is solid in its structural choices: a bottom tab bar on phones, a sticky bid bar on the auction page, a 3-step inventory wizard, dense-row truncation, and ≥44 px tap targets. The code is organised into clearly-labelled per-slice blocks at the bottom of src/styles.css. The gaps below are mostly about fluidity and robustness, not missing features.

# Finding Severity Detail
1 Typography is not fluid HIGH Only .hero h1 uses clamp(). Everything else is a fixed rem: .brand 1.05, .nav a 0.9, .stat .value 1.75, .metric 2, .card h2 0.95, .tab 0.9, .sub 0.82, badges/tags 0.7–0.72. Text looks oversized on small phones and undersized on large desktops, and there's no scale between the two.
2 Inconsistent breakpoints MED The file mixes 640, 641, 767, 768. Nav/page/dashboard use 640; tap targets and the bid bar use 768; the wizard uses 767/768. No single documented scale, so boundaries are easy to get wrong (the admin-filters bug lived exactly at one of these seams).
3 Magic-number coupling on auction detail MED The fixed bid bar reserves space with padding-bottom: 20rem on .auction-detail-page and is offset above the nav with bottom: calc(52px + safe-area). These constants are guesses; if the bid card grows (validation errors, long buyer names) content can be obscured. Better to measure the bar height into a CSS variable, or use position: sticky.
4 100vw can cause horizontal scroll MED .page uses min(1240px, calc(100vw - 2rem)) and the phone override uses calc(100vw - 1.5rem). 100vw includes the vertical scrollbar width on desktop, so on scrollbar-ful platforms the page is slightly wider than the content area, producing a faint horizontal scrollbar. Prefer 100% / 100dvw with a max-width.
5 100vh on mobile browser chrome LOW .shell { min-height: 100vh } and the bid-bar max-height: 60vh ignore the dynamic mobile toolbar. Use 100dvh/60dvh with a vh fallback.
6 Spacing is fixed, not fluid LOW Card padding (1.1rem), page padding, grid gaps, and section margins are constant. They're in rem so they're not broken, but a fluid spacing scale would make large screens feel less cramped and small screens less wasteful.
7 Responsive images not guaranteed LOW Fish images come from arbitrary external URLs. .preview-image caps at 260px, but there is no global img { max-width: 100%; height: auto } safety net for any image rendered elsewhere.
8 Layout keys off viewport, not container LOW Card-dense components (dashboard lists, admin cards) reflow only at global breakpoints. Container queries would let a card adapt to the column it actually sits in, which matters once the dashboard grid varies between 1–3 columns.
9 Scattered, duplicated media queries LOW The per-slice architecture (good for parallel work) left ~8 separate @media (max-width: 640px)/768px blocks. Now that the slices have landed, these can be consolidated for readability without changing behaviour.
Good and intentionally kept: the admin views use card lists, not <table>, so there is no wide-table overflow problem on phones. Tap targets, reduced-motion handling on the ticker, and safe-area insets on the bottom nav are already in place. The plan below builds on these rather than replacing them.

The plan

Five phases, ordered by user-visible impact. Phase 1 (fluid type) addresses the explicit request and is the highest-leverage change. Each phase is independently shippable.

Phase 1 — Fluid typography system biggest win

  • Add a modular, viewport-fluid type scale as CSS custom properties on :root in src/styles.css, e.g. --fs-xs … --fs-3xl, each a clamp(min, preferred + vw, max). Example: --fs-xl: clamp(1.35rem, 1.1rem + 1.4vw, 1.75rem);
  • Replace fixed font-size values with the tokens across the components flagged in finding 1: .hero h1/p, .stat .value/.label/.delta, .metric, .card h2/h3, .brand, .nav a, .tab, .amount, .sub, badges and tags.
  • Set an explicit base: keep the body at 16px (don't shrink form inputs below 16px — that triggers iOS auto-zoom). Verify inputs/selects/textarea stay ≥16px.
  • Respect user zoom and accessibility: use rem in the clamp min/max (not px) so OS text-scaling still works.
  • Verify in both themes and at 320px, 768px, 1440px widths.

Phase 2 — Breakpoint & spacing tokens

  • Document one canonical breakpoint set in the Stage-0 header comment and use it everywhere: sm 640, md 768, lg 1024. Resolve the 641/767/768 seams so each boundary is owned by exactly one rule (the admin-filters fix is the template: pair max-width: 640px with min-width: 641px).
  • Add a fluid spacing scale (--space-1 … --space-6 using clamp()) and apply to .page padding, .card padding, grid gaps, and section margins so density tracks viewport.
  • Introduce an lg (1024px+) tier the site currently lacks: widen the page container, allow the dashboard grid a denser 3–4 column layout, and let stats grow.

Phase 3 — Layout robustness

  • Auction detail bid bar: replace the 20rem / 52px magic numbers. Either (a) switch the bar to position: sticky; bottom: 0 inside a scroll container, or (b) measure its height in a small effect and expose it as --bid-bar-h, then use padding-bottom: var(--bid-bar-h).
  • Admin filters component: retire the CSS ::details-content workaround by rendering a media-query-driven component — always-open panel on desktop, a real toggle button + state on mobile. Removes the fragile dual-mode <details>.
  • Swap 100vw → 100%/100dvw and 100vh → 100dvh (with fallbacks) to kill the horizontal-scroll and mobile-toolbar issues (findings 4–5).
  • Add the global image safety net: img { max-width: 100%; height: auto; }.

Phase 4 — Container queries for dense components

  • Mark the dashboard/admin card grids as container-type: inline-size and move the most layout-sensitive rules (row wrapping, two-line vs one-line bid rows) from @media to @container so cards adapt to their column width.
  • Scope to the components that benefit; don't blanket-convert working media queries.

Phase 5 — Consolidation & QA

  • Merge the scattered per-slice @media blocks now that parallel development is done, keeping a short comment trail. No behavioural change — purely readability.
  • QA matrix: 320 / 375 / 414 / 768 / 1024 / 1440 px × light/dark × prefers-reduced-motion. Verify the four routes plus the shell, no horizontal scrollbars, tap targets, and that the bid bar never hides content.
  • Update the Playwright e2e suite (e2e/tests/*) to lock in the responsive behaviour:
    • Add a regression test for the admin-filters desktop fix — assert the filter fields (Status / Species / Seller) are visible at a desktop viewport without clicking the Filters summary, and that they are collapsed-by-default at a phone viewport.
    • Add mobile-viewport projects in playwright.config.ts (e.g. a Pixel/iPhone device descriptor) so the existing flows run at phone width; assert the bottom tab bar is visible and the desktop top-nav links are hidden.
    • Add a horizontal-scroll guard helper used across the four routes: scrollWidth <= clientWidth at 320px and 1440px.
    • Update the affected page objects (e2e/pages/admin.ts, auction.ts) for any selector/visibility changes from Phase 3.
  • Run pnpm lint, pnpm format:check, and pnpm test:e2e; keep changes CSS-token-driven.
