Mobile Strategy Plan

A detailed plan for making the frontend work well for mobile (phone) users, based on a review of every view page. Scope is the four view routes plus the shared shell. /metrics is a raw text endpoint, not a view, so it is out of scope.

  • Responsive breakpoints
  • Mobile navigation
  • Dense-row tuning
  • Sticky bid bar
  • Inventory wizard
  • Admin tabs polish
  • Parallel vertical slices

1. Current state (what we have today)

The app is desktop-first with fluid sizing but no real mobile adaptation:

  • The only @media rule in src/styles.css is prefers-reduced-motion. There are no width breakpoints at all.
  • Layout survives narrow screens only because of fluid primitives: .page { width: min(1240px, calc(100vw - 2rem)) }, .grid { repeat(auto-fit, minmax(280px, 1fr)) }, .stat-strip { repeat(auto-fit, minmax(190px, 1fr)) }, and hero h1 { clamp(1.8rem, 4vw, 2.8rem) }.
  • So columns do collapse to a single column on a phone, but nothing else is tuned: the top bar, dense .row items, long forms, and tap targets are all still sized for a mouse and a wide viewport.

The viewport meta tag is already present in src/routes/__root.tsx (width=device-width, initial-scale=1), so we have a correct foundation.

2. Goals & principles

  1. Mobile-first CSS layering — keep the existing fluid layout, add a small set of breakpoints to tune spacing, typography, navigation, and dense rows.
  2. Reachability — primary navigation and the primary action on each page should be reachable with a thumb.
  3. Touch targets ≥ 44×44px — nav links, tabs, row links, buttons.
  4. No horizontal scrolling of page content (the ticker marquee is the only intentional exception).
  5. CSS-first, minimal JS — prefer responsive CSS over JS layout switching to keep risk low (no Playwright in this project; see §6).

Proposed breakpoints

Add to src/styles.css:

--bp-sm: 640px;   /* large phone / small tablet */
--bp-md: 768px;   /* tablet */
--bp-lg: 1024px;  /* desktop */

Tune at max-width: 640px (phone) and max-width: 768px (tablet) since the base styles are desktop-sized.

3. Should we split pages into subpages?

Page Recommendation
/inventory/new Yes — split into a 3-step flow. Highest-value split. See §4.4.
/admin No split needed — tabs already solve it. Keep the single route; just make the tab strip horizontally scrollable and collapse filters on mobile.

4. Detailed work plan, page by page

4.1 Shared shell & navigation (__root.tsx, styles.css)

  • Convert .topbar for phones: keep brand + theme toggle on the top row, move the 4 primary links into either a (recommended) fixed bottom tab bar (Dashboard / Add fish / Admin / Metrics) for an app-like, thumb-reachable feel, or a hamburger menu that toggles a dropdown panel.
  • If bottom nav: add padding-bottom to .page equal to the bar height so content isn't hidden; respect env(safe-area-inset-bottom) for notched phones.
  • Increase nav link padding to hit 44px tap height; mark the current route with the existing .active style.
  • Make .topbar non-sticky or shorter on phones to reclaim vertical space.

4.2 Dashboard (index.tsx)

  • .stat-strip: drop to 2 columns at ≤640px, 1 column at ≤400px (override the minmax(190px,...) so tiles don't get too tight).
  • .row density fix (shared win across all pages): allow the label block to shrink and truncate, and let the amount drop to its own line if needed: .name-wrap { min-width: 0 } + overflow: hidden; text-overflow: ellipsis on the name/sub; at ≤640px allow .row { flex-wrap: wrap }.
  • Ticker: confirm the play/pause toggle tap target is ≥44px; otherwise unchanged.
  • Optional: add a "View all" link per card to future index routes.

4.3 Auction detail (auctions.$auctionId.tsx)

  • Keep the two-card stack, but on ≤768px turn the Place bid card into a sticky bottom action bar (current price + amount input + Submit) so a buyer can bid without scrolling past the live chain.
  • Give the Live bid chain list a max-height with internal scroll on mobile so the page doesn't grow unbounded and the sticky bar stays useful.
  • Ensure the buyer <select> and amount <input> are full width with ≥44px height (mostly true already via .field input).
  • Keep the SSE/refresh logic unchanged — this is a pure layout change.

4.4 Add inventory (inventory.new.tsx) — split into steps

Convert the long form into a 3-step wizard to reduce scrolling and cognitive load on a phone. Two viable implementations:

  • (Recommended) In-route stepper: keep the single /inventory/new route, add a step state (1→3) with Back/Next, validating each step before advancing. Lower routing complexity, preserves form state easily, and the existing Zod validation in validateForm can be applied per-step.
  • Route-segment subpages: /inventory/new/details, /inventory/new/pricing, /inventory/new/auction. Cleaner URLs/deep-linking but requires lifting form state into a parent layout route or a store; more work, easy to lose state on refresh. Defer unless deep-linking is needed.

Suggested step grouping:

  1. Fish — species, display name, weight (kg), catch region, freshness/grade.
  2. Pricing & seller — starting price, seller, description, image URL + preview.
  3. Auction (optional) — the existing createAuction toggle + starts/ends/minimum increment.

On desktop (≥768px) we can keep showing the full form (no stepper) so we don't regress the existing experience — the stepper is the mobile presentation.

4.5 Admin (admin.tsx) — finish the mobile tuning

Tabs already removed the long scroll. Remaining:

  • Make the .tabs strip horizontally scrollable on narrow screens (overflow-x: auto; flex-wrap: nowrap) so 4+ tabs never wrap into two rows; add momentum scrolling and hide the scrollbar.
  • Collapse AdminFilters into a single "Filters" disclosure (closed by default on mobile) to reclaim vertical space; check filter-fields.tsx grid stacks to one column.
  • Active tab panels already render only 1–2 cards, which expand to full width — good on mobile. Verify card .row action buttons (Close/Withdraw/Start auction) wrap and keep ≥44px height.

5. Shared CSS changes (one place, many wins)

Most pages benefit from a small set of shared rules in src/styles.css:

  • .row truncation + wrap behavior (see §4.2).
  • .page padding reduced at ≤640px (e.g. padding: 1.25rem 0 5rem to leave room for bottom nav).
  • Tap-target minimums for .nav a, .tab, .button, .row a.
  • Tab strip horizontal scroll.
  • Stat strip column overrides at phone widths.

6. Testing & verification (no Playwright)

This repo deliberately excludes browser automation, so verify responsively without it:

  • Manual device-emulation pass (Chrome DevTools) at 360, 390, 414, 768px for each route: nav reachable, no horizontal scroll, bid bar sticky, wizard steps validate.
  • Add Vitest component tests for any new logic (e.g., the inventory stepper's per-step validation and step navigation) — logic only, not pixels.
  • Keep changes CSS-first so the existing unit/integration tests continue to cover the domain behavior unchanged.

7. Implementation order — built for parallel work

The work is structured as one small serial foundation followed by independent vertical slices. After the foundation lands, every slice can be picked up by a different developer (or agent) at the same time, because each slice owns its own route/component files and is verifiable end-to-end on its own.

Stage 0 — Foundation (serial prerequisite, keep it tiny)

The only true shared dependency. Land this first and fast so it unblocks every parallel slice. Scope it deliberately small to minimise the window where others rebase on it.

Stage 0 — Shared CSS foundation In src/styles.css: breakpoint tokens (§2), the shared .row truncation/wrap utility, tap-target minimums, and .page padding (room for bottom nav). Add page-specific overrides under clearly labelled, per-route comment blocks (e.g. /* mobile: dashboard */) so later slices append instead of colliding. Low risk, broad benefit, no behaviour change.

Stage 1 — Parallel vertical slices (all independent, do in any order / concurrently)

Each slice is a complete, shippable mobile improvement for one surface: markup change + its scoped CSS block + any logic tests. File ownership is disjoint, so these can run simultaneously with minimal merge risk (see ownership map below).

Slice A — Navigation Mobile bottom nav / hamburger. Owns __root.tsx + its /* mobile: nav */ CSS block. Verifiable alone: nav reachable, active route marked, content not occluded.
Slice B — Admin polish Scrollable tab strip + collapsible filters. Owns admin.tsx, admin-filters.tsx, filter-fields.tsx + /* mobile: admin */.
Slice C — Auction detail Sticky mobile bid bar + scrollable live chain. Owns auctions.$auctionId.tsx + /* mobile: auction-detail */. SSE logic unchanged.
Slice D — Dashboard Stat-strip column overrides + optional "view all" links. Owns index.tsx + /* mobile: dashboard */.
Slice E — Inventory wizard In-route 3-step flow (largest change). Owns inventory.new.tsx + new stepper component + /* mobile: inventory */ + Vitest tests for per-step validation. Self-contained; does not block the others.

File-ownership map (how the slices stay conflict-free)

Slice Primary files owned Shared file touched

B — Admin admin.tsx, admin-filters.tsx, filter-fields.tsx styles.css → /* mobile: admin */ only

Only Stage 0 is sequential, and it is intentionally tiny. Because each Stage 1 slice owns disjoint route/component files and appends to its own labelled CSS block, slices A–E can be developed in parallel and merged independently. Slices A–D each land most of the mobile win on their own surface; Slice E is the only one that restructures a page, but it is still independent and can run concurrently with the rest.
